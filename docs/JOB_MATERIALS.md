# 求职材料与面试准备

## 简历项目描述

> 独立设计并实现可追溯法律分析 Agent：使用 FastAPI、LangGraph、SQLAlchemy 与 React/TypeScript 构建案件抽取、混合检索、引用审核、后台任务和报告导出闭环；以数据库 article ID 约束模型引用并提供离线降级。构建 24 条脱敏评测集和关键词/哈希向量/混合检索对照实验，离线工作流成功率 100%，案件类型规则基线经评测驱动从 75% 提升至 100%；配置 Ruff、mypy 与 GitHub Actions，32 项自动测试覆盖模型失败、损坏文件、注入文本和数据库锁重试。

指标只适用于当前教学数据集。在线 Embedding 未实际运行前，不应在简历中写真实向量模型效果。

## 30 秒项目介绍

这个项目解决大模型法律回答中“法条可能编造、结果不能回查”的问题。我把流程拆成案件要素提取、法规检索、引用审核和报告生成四个 LangGraph 节点。每条引用都必须关联数据库 article ID 并通过原文一致性校验；模型不可用时可以降级到规则方案。项目还有 24 条脱敏评测集、检索对照实验、32 项测试和 GitHub Actions CI。

## 高频面试问题

### 为什么需要 Agent，而不是一次模型调用？

一次调用把抽取、检索、校验和写作混在一起，无法分别测试或重试。Agent 工作流的价值是显式状态、条件分支、工具调用和失败恢复，不是类名里出现 Agent。

### Agent 与普通流水线有什么区别？

当前前四个节点大部分是确定流程，但引用不足时会根据状态回到检索节点；未来可以加入人工审批。若永远是固定顺序，普通流水线更简单，不应为框架而框架。

### 如何减少法条幻觉？

模型只能使用检索得到的 article ID；服务端再核对 ID、名称、条号和正文；报告只接收审核通过的引用。它减少“引用编造”，但仍需权威来源核对现行有效性。

### RAG 如何评测？

固定脱敏数据集和相关 article ID，计算 Recall@5、MRR、Hit@K 与 Precision@5，同时记录延迟。抽取计算案件类型准确率和关键词覆盖率，系统层统计成功率、平均/P95 耗时和模型调用成本。

### 为什么选择 Top-K=5？

法规库只有 10 条，K=5 能观察召回同时限制报告上下文。K 不是固定真理，应在扩大数据集后绘制 K 与召回、精度、延迟和上下文成本曲线。

### 如何处理模型失败？

认证错误立即失败该次调用；限流和服务端错误有限重试；JSON 结构错误只修复一次；之后回退规则/模板并记录原因。CI 完全不访问真实模型。

### 如何保护上传文件？

前端校验用于体验，后端独立限制大小、扩展名、MIME、空文件和解析超时；文件名只作安全展示，不拼接路径；内容不进入 SQL 或系统命令；当前版本不持久保存上传原文件。

### 如何迁移到生产环境？

使用 PostgreSQL/pgvector、对象存储、持久任务队列、鉴权与租户隔离；加入法规版本同步、审计日志、指标告警和数据保留策略；通过影子流量与人工标注集验证迁移前后效果。
