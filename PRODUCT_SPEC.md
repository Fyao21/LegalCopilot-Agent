# 律镜 Legal Copilot Agent 产品说明书

## 1. 产品信息

- 产品名称：律镜 Legal Copilot Agent
- 当前版本：0.3.0（第三周版本）
- 目标用户：希望快速整理合同纠纷或劳动争议材料的个人、法律科技学习者和项目评审者
- 产品性质：法律 AI 工程演示项目
- 风险声明：所有输出仅供技术演示，不构成法律意见，法规必须以权威来源的现行有效文本为准

## 2. 用户问题

用户拿到案件材料时，通常需要完成三件事：从长文本中整理事实；找到可能相关的法律条文；将事实、争议和依据组织为便于阅读的分析。直接让大模型回答容易出现虚构条文、引用无法回查和执行过程不可见的问题。

本产品通过“结构化提取、知识库检索、引用审核、报告生成”四个节点解决这些问题。模型只负责适合模型的理解和表达，条文真实性由数据库 ID 和服务端校验负责。

## 3. 当前版本范围

### 已支持

1. 上传 TXT、DOCX、带文本层的 PDF，或只输入案件问题。
2. 提取案件类型、当事人、事实、诉求、争议焦点、信息缺口和待确认问题。
3. 支持合同纠纷和劳动争议两类教学场景。
4. 使用关键词分数与语义分数进行混合检索。
5. 使用 LangGraph 执行案件分析、法规检索、引用审核和报告生成。
6. 引用审核会校验条文 ID、法律名称、条号和正文是否与数据库一致。
7. Agent 模式可以调用 OpenAI 兼容的模型接口；模型不可用时自动回退到规则方案。
8. 离线模式不需要模型 API Key，能够完整生成 Markdown 报告。
9. 检索失败时最多补充检索两次，之后生成明确标注的低置信度报告。
10. 提供任务状态、节点轨迹、引用列表和报告读取接口。
11. 提供 React + TypeScript 浏览器界面，支持拖拽上传、示例问题、进度轮询和错误提示。
12. Agent 工作流通过 FastAPI 后台任务执行，创建请求立即返回 `queued`。
13. 支持 Markdown 和中文 PDF 报告下载。
14. 上传文件限制为 TXT、DOCX、PDF 和 10 MB，并校验扩展名、MIME、空文件与解析超时。
15. 请求日志包含 request ID、路径、状态码和耗时，不记录密钥与案件全文。
16. 提供前后端 Dockerfile 和 Docker Compose 一键启动配置。

### 当前不支持

1. 扫描版 PDF OCR。
2. 跨进程持久任务队列；当前使用 FastAPI 进程内后台任务，进程退出后未完成任务不会自动恢复。
3. 全量权威法规库和法规版本自动更新。
4. 用户账号、权限、计费和多租户。
5. 真实法律意见、胜诉预测或自动诉讼决策。

## 4. 两种运行模式

### Offline 模式

请求参数 `mode=offline`。使用确定性规则提取案件，使用中文二元词哈希向量和关键词混合检索，使用模板生成报告。

适用场景：本地开发、自动测试、没有 API Key、演示降级能力。

### Agent 模式

请求参数 `mode=agent`。当 `.env` 中 `OFFLINE_MODE=false` 且配置了模型密钥时，案件分析、语义审核和报告写作使用 LLM；Embedding 可单独配置真实服务。

如果模型调用超时、输出不符合 Schema 或没有密钥，系统记录回退原因并继续使用规则或模板，不让单次模型故障导致整个服务退出。

## 5. 核心工作流

```text
analyze_case
    ↓
retrieve_laws
    ↓
review_citations
    ├─ 有可信引用 → write_report → END
    ├─ 无可信引用且重试次数 < 2 → retry_retrieval → retrieve_laws
    └─ 无可信引用且达到上限 → 低置信度 write_report → END
```

### 节点职责

- `analyze_case`：抽取结构化案件事实；LLM 失败时回退规则。
- `retrieve_laws`：构造检索问题，融合关键词与语义相似度。
- `review_citations`：先做数据库一致性校验，再做可选语义支撑审核。
- `retry_retrieval`：增加案件类型、法律责任和请求权基础等检索信息。
- `write_report`：只使用案件事实和审核通过的引用生成报告。

## 6. 主要数据

### 法规

每条法规保存 `article_id`、法律名称、条号、正文和来源。不同 Embedding provider/model 的向量单独存放，记录内容哈希，正文不变时不会重复计算。

### Agent 任务

任务保存原始问题、文件文本、运行模式、状态、进度、重试次数、案件要素、节点轨迹、报告、错误和模型名称。

### 审核引用

每条引用保存综合分数、关键词分数、语义分数、审核状态、审核理由和是否通过。报告只展示通过审核的条文。

## 7. 主要接口

- `GET /health`：健康检查。
- `POST /api/v1/articles/search`：直接检索法规。
- `GET /api/v1/articles/{article_id}`：回查法规原文。
- `POST /api/v1/cases`：第一周兼容的同步分析接口。
- `POST /api/v1/runs`：执行完整 Agent 工作流。
- `GET /api/v1/runs/{run_id}`：读取状态和节点轨迹。
- `GET /api/v1/runs/{run_id}/citations`：读取审核结果。
- `GET /api/v1/runs/{run_id}/report`：读取 Markdown 报告。
- `POST /api/v1/runs/{run_id}/retry`：重试失败任务并创建新任务。

完整请求与响应见 `docs/openapi.json`。

## 8. 成功标准

第二周版本达到以下标准才算完成：

1. 无 API Key 时完整工作流成功结束。
2. 合同和劳动问题能够识别正确领域。
3. 每条报告引用都能通过 `article_id` 回查数据库原文。
4. 篡改引用正文后审核节点将其拒绝。
5. 模型输出非法 JSON 时只修复一次，仍失败则安全回退。
6. 空知识库最多重试两次，不发生无限循环。
7. 所有自动测试通过。
