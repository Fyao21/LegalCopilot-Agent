# 律镜 Legal Copilot Agent：项目目标与四周执行手册

## 1. 文档用途

这不是一份只描述方向的路线图，而是一份可以逐项执行和验收的开发手册。每个任务都包含“为什么做、具体怎么做、需要产生什么文件、如何确认完成”。完成任务后，在标题前标记状态：

- `[ ]`：尚未开始；
- `[~]`：正在进行；
- `[x]`：已经完成并通过验收；
- `[!]`：遇到阻塞，需要记录原因和下一步处理方式。

每天下班前需要做三件事：运行自动测试、更新当天的任务状态、记录当天发现的问题。不要在测试失败的情况下开始下一阶段。

## 2. 项目最终目标

项目名称：律镜 Legal Copilot Agent。

项目定位：面向合同纠纷和劳动争议的可追溯法律分析助手。用户上传 TXT、DOCX 或 PDF 案件材料并输入问题，系统抽取案件要素，从自己的法律知识库中检索相关条文，审核引用是否真实有效，然后生成包含依据、出处和免责声明的分析报告。

最终用户流程：

```text
上传案件材料和问题
        ↓
解析文件并清洗文本
        ↓
提取当事人、事实、诉求和争议焦点
        ↓
生成检索问题并搜索法规知识库
        ↓
审核法条引用和分析结论的一致性
        ↓
生成 Markdown/PDF 报告
        ↓
查看运行过程、引用来源和质量指标
```

项目要证明的工程能力：

1. 能将模糊业务需求拆成明确的数据模型、API 和工作流节点。
2. 能处理文档输入、数据库持久化、检索、模型调用和结构化输出。
3. 能控制大模型幻觉，而不是直接相信模型生成的法条和结论。
4. 能用测试集、指标和对照实验说明效果。
5. 能让其他人在新电脑上独立启动、测试和演示项目。

## 3. 全局技术约束

- Python 版本使用 3.11 或更高版本。
- 后端使用 FastAPI，输入和输出使用 Pydantic Schema。
- 第一周默认使用 SQLite，确保无需外部服务即可运行。
- 第二周加入真实 Embedding 和 LangGraph，同时保留离线模式。
- 第三周引入 React + TypeScript 前端和后台任务。
- 第四周完成评测、CI、Docker 和求职材料。
- 所有报告引用必须包含知识库中的 `article_id`，不得只保存模型生成的法条字符串。
- LLM 的输出一律视为不可信数据，必须经过 Schema、数据库和业务规则校验。
- 用户上传内容不得直接拼接为数据库语句、文件路径或系统命令。
- 项目输出必须显示“技术演示，不构成法律意见”的声明。

### 第 3 天：案件文件解析

#### [x] 步骤 1：实现 TXT、DOCX 和 PDF 解析

目的：把不同文件格式统一转换为纯文本，交给后续分析流程。

具体操作：

1. TXT 使用 UTF-8 解码，并对错误字符执行安全替换。

4. 去掉空段落并统一换行。
5. 对 `.exe`、`.zip`、旧版 `.doc` 等不支持格式返回明确错误。
6. 不在第一周处理扫描件 OCR，记录为已知限制。

产出文件：`app/services/document_parser.py`。

验收方法：分别上传 TXT、DOCX、带文本层 PDF 和非法扩展名文件。

完成标准：三种支持格式能返回文本；非法文件返回 HTTP 415；错误不会导致服务退出。

### 第 4 天：案件要素提取基线

#### [x] 步骤 1：定义结构化案件 Schema

目的：后续无论使用规则还是 LLM，都返回同一种稳定结构。

具体操作：

2. 定义当事人列表 `parties`。
3. 定义关键事实列表 `key_facts`。
4. 定义诉求列表 `claims`。
5. 定义争议焦点列表 `dispute_focuses`。

产出文件：`app/schemas.py`。

完成标准：任意输入都能生成合法 `CaseFacts` 对象。

#### [x] 步骤 2：实现确定性规则基线

目的：没有模型密钥时仍能演示，并为第二周 LLM 方案提供对照组。

具体操作：

1. 根据“劳动、工资、加班”等词识别劳动争议。
2. 根据“合同、违约、货款”等词识别合同纠纷。
3. 使用正则表达式提取原告、被告、申请人、甲方和乙方。
4. 按句号、分号和换行切分事实。
5. 使用“请求、赔偿、支付、返还、解除”等词筛选诉求。
6. 使用“争议、是否、违约、责任、解除”等词筛选争议焦点。

产出文件：`app/services/case_analyzer.py`。

完成标准：合同样例识别为合同纠纷，工资样例识别为劳动争议；空字段安全返回。

### 第 5 天：离线检索基线

#### [x] 步骤 1：实现中文哈希向量

目的：无需下载模型也能打通检索流程，并建立可测试的基线。

具体操作：

1. 去除空白并统一为小写。
2. 将文本切分为中文二元字符片段。
3. 使用 SHA-256 将片段稳定映射到固定维度。
4. 对向量执行 L2 归一化。
5. 使用余弦相似度比较问题和条文。
6. 对维度不一致的向量抛出明确异常。

产出文件：`app/services/embeddings.py`。

完成标准：相同文本相似度接近 1；合同问题与合同条文的分数高于劳动条文。

#### [x] 步骤 2：实现 Top-K 法规检索

目的：为案件返回带数据库 ID 和出处的候选法律依据。

具体操作：

1. 将查询文本转换为向量。
2. 从数据库读取法规及其向量。
3. 计算每条法规的余弦相似度。
4. 按分数从高到低排序。
5. 截取用户指定的 Top-K，限制为 1–20。
6. 返回条文 ID、法律名称、条号、正文、来源和分数。

产出文件：`app/services/retriever.py`。

完成标准：检索结果数量符合 limit；每条结果都能用 `article_id` 回查数据库。

第二周目标：把规则基线升级为“LLM 结构化分析 + 真实语义检索 + 引用审核 + 报告生成”的可控工作流，同时保留第一周离线模式。

### 第 9 天：法规切片和真实向量入库

#### [x] 步骤 1：定义切片规则

#### [x] 步骤 2：执行向量化

1. 找出未生成真实向量或正文已变化的记录。
2. 批量调用 Embedding Adapter。
3. 校验返回数量和维度。
4. 失败批次记录原因并允许断点继续。
5. 保存 embedding provider、模型名和内容哈希。

完成标准：重复运行不会重复计算未变化的数据；能够统计成功、跳过和失败数量。

对法律名称、条号和正文进行关键词匹配；条号精确命中给予较高权重；过滤无关领域。

### 第 11 天：结构化案件分析 Agent

#### [x] 步骤 1：设计提示词和 Schema

提示词必须要求模型只根据材料提取，不允许补充不存在的事实。Schema 增加事实置信度、缺失信息和需要用户确认的问题。

#### [x] 步骤 2：调用并校验模型输出

1. 对输入文本设置最大长度。
2. 调用结构化输出接口。
3. 使用 Pydantic 校验。
4. 解析失败时允许一次格式修复。
5. 仍然失败时回退到第一周规则基线。

### 第 12 天：LangGraph 工作流

#### [x] 步骤 1：定义 `WorkflowState`

至少包含：`run_id`、原始文本、用户问题、案件事实、检索查询、候选引用、审核结果、重试次数、报告、错误和节点轨迹。

#### [x] 步骤 2：实现四个节点

1. `analyze_case`：抽取案件要素。
2. `retrieve_laws`：生成查询并执行混合检索。
3. `review_citations`：检查引用。
4. `write_report`：生成报告。

每个节点只承担一个明确职责，输入和输出必须更新 State，不能依赖隐藏的全局变量。

#### [x] 步骤 3：定义条件边

- 审核通过：进入报告生成；
- 审核不通过且重试次数小于 2：修改查询后重新检索；
- 超过重试上限：生成低置信度报告并明确提示；
- 系统错误：进入失败结束节点。

完成标准：打印或可视化工作流图时，节点、条件和结束路径清晰可解释。

### 第 13 天：引用审核与报告生成

#### [x] 步骤 1：实现确定性引用审核

逐条检查：`article_id` 是否存在；法律名称、条号和正文是否与数据库一致；正文是否为空；来源是否存在。

#### [x] 步骤 2：实现语义支撑审核

让模型判断“结论是否被引用条文支持”，但模型只能返回判断和理由，不能修改数据库原文。低置信度引用应移除或标注。

#### [x] 步骤 3：生成 Markdown 报告

报告固定包含：免责声明、案情摘要、关键事实、争议焦点、法律分析、行动建议、证据缺口、引用附录和生成元数据。

所有正文引用使用 `[1]`、`[2]` 编号，附录将编号映射回 `article_id` 和原文。

完成标准：报告中不存在无法回查的引用；删除数据库条文后审核测试会失败。

### 第 14 天：第二周集成测试与验收

1. 正常合同纠纷完整流程。
2. 正常劳动争议完整流程。
3. 模型返回非法 JSON。
4. 模型超时并触发回退。
5. 检索无结果。
6. 审核发现伪造条文。
7. 审核两次不通过后正常结束。
8. 离线模式运行第一周功能。

第二周完成标准：每条报告引用都可回查；无 API Key 时不影响离线功能；任何错误都不会产生无限循环。

## 8. 每日开发固定流程

每天开始：

1. 查看本文件中当天任务和上次遗留问题。
2. 启动虚拟环境并确认分支。
3. 运行现有测试，确保基线正常。
4. 将当天第一个任务标记为 `[~]`。

开发过程中：

1. 一次只实现一个可验收的小步骤。
2. 先写或更新测试，再修改业务代码。
3. 外部模型和数据库异常必须有明确处理。
4. 新配置同步到 `.env.example`，不得提交真实密钥。
5. 新接口同步更新 OpenAPI 描述和自测文档。

每天结束：

1. 运行完整自动测试。
2. 手工执行当天新增的关键流程。
3. 更新任务状态、已知限制和测试结果。
4. 检查 `git diff`，确保没有数据库、密钥、日志或用户材料。
5. 提交一个职责明确的 commit，提交信息说明完成了什么。

## 9. 完成定义（Definition of Done）

一项功能只有同时满足以下条件，才能标记为 `[x]`：

1. 业务行为已经实现，不是空函数、Mock 或只写 TODO。
2. 正常输入和至少一个异常输入有测试。
3. 测试在本地通过，没有因为顺序不同而失败。
4. API 输入输出有 Schema，错误信息对用户可理解。
5. 日志不泄露密钥、完整上传文本和敏感数据。
6. README 或自测文档已补充操作方法。
7. 没有引入对旧项目的运行时依赖。
8. 代码能够由另一个人根据文档复现。

## 10. 风险和处理方案

### 模型输出不稳定

使用结构化输出、Pydantic 校验、有限重试和规则回退。不得通过无限重试掩盖问题。

### 法条数据不权威

教学样例明确标注；正式数据记录来源 URL、版本和生效日期；报告始终提示核验权威来源。

### 检索指标不理想

先检查标注质量，再分别分析分词、Embedding、Top-K、权重和领域过滤。每次只调整一个变量并重新跑相同评测集。

### 项目范围膨胀

MVP 只支持合同纠纷和劳动争议。用户登录、计费、全案由、大规模分布式部署不进入四周核心范围。

### 长任务影响体验

第三周拆为后台任务，返回 run ID；前端轮询；生产方案在技术文档中说明持久任务队列。

### 求职展示只剩界面

README 必须同时展示架构、工作流、测试、评测和失败处理。面试演示既展示用户界面，也展示引用回查和测试指标。
